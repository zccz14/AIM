# 为 Task 结果上报补充专用 resolve / reject 终态接口

## Assumptions

- 当前服务端已经存在 `PATCH /tasks/{id}`，并沿用 `TASK_VALIDATION_ERROR`、`TASK_NOT_FOUND` 等既有错误模式处理任务更新请求。
- 当前 `Task` 持久化模型已经承载生命周期字段；本次可以在同一模型中直接新增持久化字段 `result`，不需要维持旧数据兼容。
- 目前任务终态上报仍依赖通用 PATCH 更新多个字段；调用方需要自己组合 `status` 与附加结果文本，接口语义偏泛化。
- 现有任务创建流程允许为新持久化字段提供默认值，因此 `result` 可以在创建时统一初始化为空字符串，而不要求调用方显式传入。
- 当前 PATCH 语义支持“未提供字段则不变更”，因此只要补充 `result` 为可选更新字段，就可以保持既有部分更新行为不退化。

## Goal vs Non-Goal

### Goal

- 新增两个专用终态接口：`POST /tasks/{task_id}/resolve` 与 `POST /tasks/{task_id}/reject`。
- 两个接口都要求 JSON body 中提供必填且非空字符串字段 `result`，用于记录任务结果文本。
- `resolve` 只负责把任务置为 `succeeded` 并写入 `result`；`reject` 只负责把任务置为 `failed` 并写入 `result`。
- 两个接口成功时统一返回 `204 No Content`，强调这是结果上报动作而不是资源读取动作。
- 在 `Task` 持久化模型中新增非空 `result` 字段，数据库列定义为 `TEXT NOT NULL DEFAULT ''`，并让任务创建默认写入空字符串。
- 保持现有 `PATCH /tasks/{id}` 仍可用；当 PATCH 请求未包含 `result` 时，已有 `result` 值必须保持不变。
- 专用接口的路径参数、请求体验证和任务不存在处理继续沿用既有 `TASK_VALIDATION_ERROR` / `TASK_NOT_FOUND` 模式，避免引入新的错误协议。

### Non-Goal

- 不把本次扩展为新的通用状态流转框架；仅覆盖任务终态结果上报的两个最小专用入口。
- 不新增除 `result` 之外的任务持久化字段，也不重构现有任务模型的其他生命周期字段。
- 不改变现有 `PATCH /tasks/{id}` 的可用性，也不要求所有调用方立即迁移到新接口。
- 不为旧数据补写历史 `result` 内容；既然已明确无需向后兼容，则只要求 schema 与新写入路径自洽。
- 不把 `result` 放宽为可空、可缺省或空白字符串；专用接口必须拒绝空结果文本。
- 不顺带调整现有错误码命名、响应包结构或任务不存在时的处理契约。

## Core Path

当前任务生命周期已经有通用 `PATCH /tasks/{id}` 入口，但“汇报任务已成功/失败并附带结果文本”是一个高频且语义非常明确的动作。继续复用泛化 PATCH，会把“终态结果上报”与“普通字段局部更新”混在一起，调用方必须自己决定该写什么状态、该不该带结果文本，也让服务端缺少一个直接表达“现在正在上报最终结果”的稳定契约。本次增量的核心价值，就是把这个高频终态动作收敛成两个语义直达的专用入口。

主路径应保持最小而直接。服务端新增 `POST /tasks/{task_id}/resolve` 与 `POST /tasks/{task_id}/reject` 两条路由，两者都只接受一个结构极小的请求体：必填非空字符串 `result`。其中 `resolve` 代表“任务已成功完成并产出结果”，因此服务端在单次请求内同时完成两件事：把任务状态写成 `succeeded`，并把 `result` 持久化到 Task；`reject` 则以完全对称的方式把状态写成 `failed` 并记录 `result`。成功后统一返回 `204 No Content`，让接口语义聚焦在“状态已被接受并落库”。

为了让这两个专用入口成为稳定契约，`Task` 本身需要拥有持久化 `result` 字段，而不是把结果文本停留在瞬时请求里。该字段应设计为非空字符串模型，数据库列使用 `TEXT NOT NULL DEFAULT ''`，这样新建 Task 时天然有稳定初值，读写路径也不需要再处理三态语义。由于本次已明确不要求旧数据兼容，因此不需要围绕历史空值设计过渡逻辑；重点是让 schema 从现在开始对新接口和后续 PATCH 路径保持一致。

现有 `PATCH /tasks/{id}` 仍然保留，因为它承担的是更广义的任务局部更新职责，不应被本次专用终态接口替代掉。这里的关键约束不是“禁止 PATCH 写 result”，而是保证 PATCH 的部分更新语义不退化：当 payload 省略 `result` 时，已有值保持不变；只有显式提供 `result` 时才更新它。这样专用接口负责简化最常见的终态上报路径，通用 PATCH 继续承担其他更新需求，二者职责清晰但不互相冲突。

备选路径之一是继续只用 `PATCH /tasks/{id}`，通过约定让调用方手动同时传 `status` 和 `result`。这条路径实现表面更省事，但没有真正把终态上报建模成独立契约，仍然把状态组合正确性的负担留给调用方，也无法从接口层直接表达“resolve 与 reject 是对称且受约束的两个动作”。另一个备选路径是新增单一 `POST /tasks/{id}/result` 再由请求体区分成功或失败，但这样会重新把“终态类型”塞回请求参数，反而弱化了动作语义。相较之下，拆成 `resolve` / `reject` 两个对称端点，语义最直接，调用也最简单。

## Value Alignment

- 当“让终态上报接口语义更直接”和“尽量复用现有泛化 PATCH”冲突时，优先前者；因此新增专用 `resolve` / `reject`，而不是继续要求调用方手动组合 PATCH 字段。
- 当“保持持久化模型简单稳定”和“为旧数据兼容增加额外过渡逻辑”冲突时，优先前者；因此 `result` 直接落为 `TEXT NOT NULL DEFAULT ''`，不为历史数据额外设计兼容层。
- 当“结果文本必须可依赖存在”与“允许 `null`、缺省或空白字符串以放宽调用约束”冲突时，优先前者；因此专用接口必须校验 `result` 为必填非空字符串。
- 当“新增专用终态入口”与“破坏现有 PATCH 局部更新能力”冲突时，优先兼容现有更新路径；因此 PATCH 继续保留，并在省略 `result` 时保持原值不变。
- 当“尽量少引入新协议”与“为新端点单独发明错误模型”冲突时，优先前者；因此验证失败和任务不存在继续复用既有 `TASK_VALIDATION_ERROR` 与 `TASK_NOT_FOUND` 模式。
