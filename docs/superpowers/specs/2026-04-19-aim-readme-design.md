# AIM 初始项目范围与 README 重写设计说明

## 背景 / 问题

当前仓库根目录中的 `README.md` 仍然主要承载 `CZ-Stack` 模板语义，不再适合作为 `AIM` 的对外入口。随着项目定位逐步清晰，仓库需要一份新的 README，用来回答三个更关键的问题：

1. `AIM` 到底是什么，而不再只是“从哪个模板演化而来”。
2. 它首先为谁服务，以及它解决的是哪一类一阶问题。
3. 它为什么要刻意保持强约束、强观点，而不是一开始就做成一个完全通用的平台。

如果 README 继续沿用模板导向的结构，读者会误以为：

- `AIM` 只是 `CZ-Stack` 的一个示例仓库，而不是独立产品。
- 它的主要读者是前线开发者，而不是需要放大 AI 研发杠杆的决策者或组织负责人。
- 它的核心价值是“提供模板能力”或“提供开发工具链”，而不是提升多 Agent 研发系统的吞吐、可控性与无人值守闭环能力。

因此，本次设计要先明确 `AIM` 的项目范围、产品边界与 README 的信息组织方式，为后续 README 重写提供统一依据。

## 目标

1. 明确 `AIM` 是独立产品，不再把根 README 视为 `CZ-Stack` 模板 README 的延续。
2. 明确新 README 的第一目标读者是希望提升 AI 效率杠杆的决策者，而不是前线开发者优先。
3. 明确 `AIM` 的一阶问题定义、产品定位、核心价值排序与系统形态。
4. 明确 `AIM` 与 `OpenCode` 的集成边界，避免后续 README 叙述越界到插件、hook 或 skill 注入方向。
5. 明确 MVP 初始项目范围、近期路线图与非目标，为后续 README 文案和仓库演进提供稳定边界。
6. 给出推荐 README 章节结构，使后续改写时可以直接按统一骨架展开。

## 非目标

1. 本次不直接重写 `README.md` 正文。
2. 本次不改动现有根 README，也不执行重命名；仅定义后续 README 重写的设计依据。
3. 本次不扩展为完整产品实现方案、数据库设计、接口字段设计或 GUI 信息架构设计。
4. 本次不把 `AIM` 定义为通用 AI 平台、企业协作平台或通用插件平台。
5. 本次不引入 `OpenCode` 插件、skill 注入、event hook 等侵入式集成方向。

## 产品定义

### 1. AIM 是独立产品

`AIM` 应被定义为一个独立产品，而不是 `CZ-Stack` 模板 README 的继续包装。后续仓库层面的表达应体现为：

1. 现有根 `README.md` 会在后续重构中退出主入口角色，并改名为 `CZ-Stack.README.md`。
2. 新的根 README 只服务于 `AIM` 的定位阐述，不再承担模板介绍、模板使用说明或脚手架导向职责。
3. `CZ-Stack` 在 README 中应退居为项目来源背景，而不是当前产品主叙事。

### 2. 第一目标读者

新 README 的第一目标读者不是“准备立刻 clone 仓库并本地开发的工程师”，而是以下角色：

1. 需要决定是否投入和采用多 Agent 研发体系的决策者。
2. 需要提升 AI 研发效率杠杆、吞吐与管理半径的负责人。
3. 需要理解 `AIM` 为什么存在、它解决什么问题、以及它不解决什么问题的产品/技术负责人。

这意味着 README 的开篇必须优先回答“为什么需要 `AIM`”与“它如何改变研发组织的效率边界”，而不是优先展开安装、开发命令或仓库目录说明。

### 3. 第一性原理问题定义

`AIM` 的 README 需要从第一性原理出发定义问题：

1. 在大规模 AI 开发中，最先成为瓶颈的不是代码生成速度，而是人的认知带宽。
2. 当并行 Agent 数量上升时，人类无法继续保持对每个实现细节的强控制。
3. 同时，组织的时间预算有限，系统必须在有限时间内完成更多基线迭代。

因此，`AIM` 要解决的问题不是“如何让单个 Agent 写代码更聪明”，而是：

- 如何在人类放弃细节控制的前提下，仍然提高整体研发吞吐。
- 如何让更多基线迭代被持续完成，而不是卡在人类同步、审阅和调度上。
- 如何让系统向更高程度的无人值守闭环执行演进。

### 4. 产品定位

`AIM` 的定位应明确为：**面向 AI Agent 的 Manager，是 Multi-Agent 研发轨道上的执行调度与编排层。**

这个定位包含三层含义：

1. `AIM` 的主要管理对象不是传统人类成员，而是执行研发任务的 Agent 会话与任务闭环。
2. `AIM` 不以“聊天入口”或“IDE 增强”作为核心产品形态，而以“调度、推进、观察、闭环”作为中心。
3. 它的长期愿景是走向 `lights-out factory`，即尽可能减少人工盯盘、人工串联与人工推进。

## 核心价值排序

README 应明确以下价值优先级，而不是平均表述多个好处：

1. **第一优先级是吞吐。** `AIM` 首先追求单位时间内完成更多有效基线推进。
2. **第二优先级是可控性。** 在吞吐提升的同时，系统仍需保持任务状态、执行边界和出站结果可观察、可追踪。
3. **第三优先级是无人值守闭环。** 系统应尽量减少人为接力，使 Task 从进入系统到完成基线推进之间更接近自动闭环。

README 不应把 `AIM` 叙述成“主要为了安全感和可视化而存在”；这些能力重要，但属于支撑更高吞吐和闭环能力的配套价值。

## 为什么 AIM 必须是有观点的

`AIM` 不追求“完整通用性”，这不是能力不足，而是刻意选择。README 需要单独解释这一点。

### 1. 有观点来自问题边界，而不是风格偏好

`AIM` 面对的是一个非常具体的问题域：如何在多 Agent 研发中，把基线持续向目标状态推进。这个问题域天然要求系统做出明确取舍，而不是保持抽象中立。

### 2. AIM 的优势来自强约束

`AIM` 的优势不是“什么都能接”，而是以下强约束的组合：

1. 单线研发推进，而不是面向任意团队工作流的通用协同抽象。
2. 强 GitHub 集成，而不是同时对齐大量代码托管和审批后端。
3. 强 `OpenCode` API 集成，而不是支持任意 Agent 宿主的最大兼容层。
4. 以基线推进为中心的调度模型，而不是通用 ticket workflow 引擎。

这些约束让系统可以围绕“如何更快地把基线推进一步”进行深度优化，而不是被大量泛化需求稀释。

### 3. 为什么不走大公司式通用平台路线

README 需要明确说明：`AIM` 当前不打算模仿大公司内部通用平台。

原因不是这些能力不重要，而是它们对应的是不同阶段、不同资源条件下的问题：

1. 大公司平台通常需要首先服务多组织、多权限模型、多类工作流与多宿主兼容性。
2. `AIM` 当前要争取的不是横向适配面，而是纵向闭环深度。
3. 在项目早期，如果先投入复杂权限、插件市场、计费、泛工作流编排等重平台能力，核心价值会被延后，系统也会更难形成清晰产品优势。

因此，README 应把“有观点”表述为产品策略，而不是临时妥协。

## 系统形态

README 后续重写应把 `AIM` 的系统形态定义为以下四个部分：

1. **独立 Server**：承载任务接入、调度推进、状态协调等核心后端职责。
2. **独立 GUI**：用于展示任务、执行过程、PR 状态与系统推进情况，但不与核心调度逻辑耦合成单体。
3. **独立 OpenAPI 规范**：作为统一对外契约，支撑 CLI、GUI 与外部集成。
4. **全局可安装 CLI**：作为进入系统的轻量入口，服务任务提交、查询、调度触发等交互。

这里的重点不是“组件越多越好”，而是明确 `AIM` 是一套独立产品形态，而不是某个单一仓库脚本或 README 中的概念集合。

## 与 OpenCode 的集成边界

`AIM` 与 `OpenCode` 的关系应被清晰定义为：**强集成，但仅通过 API。**

README 需要明确以下边界：

1. `AIM` 会强依赖 `OpenCode` 的 API 集成能力，以连接 Agent 执行层。
2. `AIM` 不会把自己设计成 `OpenCode` 插件。
3. `AIM` 不会依赖 skill 注入。
4. `AIM` 不会依赖 event hook 集成。
5. `AIM` 不应在 README 中暗示“未来大概率会作为 `OpenCode` 内置能力出现”。

这样做的目的，是保持 `AIM` 作为独立编排层的边界清晰：它可以深度利用 `OpenCode`，但不把自己的产品定义缩成对方内部扩展机制的一部分。

## 调度哲学

README 需要吸收 `docs/scheduler.md` 中已经确立的调度哲学，并用更面向外部读者的语言表达。以下内容应保持一致，不得与现有调度文档冲突。

### 1. Task 的定义

`Task` 不是普通待办事项，而是一次**基线迭代增量**。它的职责是把当前基线向目标状态推进一步。

### 2. 调度器的目标

调度器只有一个目标：**尽快把基线推进到目标状态。**

这意味着 README 不应把调度器描述成通用的任务看板引擎或复杂审批流平台，而应强调基线推进、并行度和最小人工介入。

### 3. Task 成功的判定

README 在后续表达中必须与现有调度定义一致：Task 的成功不等于“代码写完了”或“PR 开出来了”，而是至少满足以下闭环：

1. PR 已合并。
2. 对应 worktree 已清理。
3. 主工作区基线已刷新。

只有这三者同时成立，才意味着一次基线推进真正完成。

### 4. 调度中心思想

README 中应传达以下中心思想：

1. 系统关注的是基线是否前进，而不是任务列表是否被人工更新。
2. 调度的核心对象是可被闭环推进的 Task，而不是等待持续人工确认的步骤。
3. 成功的系统应逐步减少人工盯盘和人工接力。

## 初始项目范围（MVP）

README 重写后的“初始范围”章节应聚焦于以下 MVP 能力，不提前承诺更重的平台能力：

1. **任务接入与调度**：允许外部把 Task 交给系统，并由系统负责推进与调度。
2. **执行过程展示**：提供对任务执行进度、状态与关键结果的可见性。
3. **统一 OpenAPI 契约**：明确 Server、GUI、CLI 与外部集成围绕同一契约协作。
4. **CLI 入口**：提供一个轻量、可全局安装的命令行入口。
5. **非侵入式 `OpenCode` API 集成**：通过 API 与 Agent 执行层衔接，而不侵入 `OpenCode` 内部机制。

这里需要强调“初始范围”而不是“最终产品全貌”。README 应让读者看到一个清晰、可落地的第一阶段系统，而不是被宽泛愿景淹没。

## 近期路线图

README 后续的路线图章节应只承接当前定位自然延伸出的三类增强方向：

1. 更强的编排能力。
2. 更强的治理与可观测性。
3. 更强的无人值守闭环自动化。

路线图表达应保持方向性，不提前承诺具体时间表、组织级能力包或复杂平台模块。

## 非目标

README 需要显式列出非目标，以保护项目边界，避免外部读者按通用平台预期理解 `AIM`：

1. 不做 `OpenCode` 插件。
2. 不做 skill 注入或 event-hook 集成。
3. 不尝试成为完全通用的 AI 任务平台。
4. 不在早期承诺复杂组织权限模型。
5. 不在早期承诺插件市场。
6. 不在早期承诺计费系统。
7. 不在早期承诺大而全的平台治理能力。

这些非目标不是永久否定未来可能性，而是当前阶段的明确边界。README 表达时应避免出现“以后什么都可以加”的松散措辞。

## README 推荐结构

后续 README 重写应按照以下顺序组织内容：

1. 标题 + 一句话定位。
2. 为什么需要 `AIM`。
3. 项目立场 / 为什么不做完全通用。
4. 核心价值。
5. 系统形态。
6. 与 `OpenCode` 的集成边界。
7. 调度哲学。
8. 初始项目范围。
9. 近期路线图。
10. 非目标。
11. 来自 `CZ-Stack` 的项目起源。
12. 面向开发者的仓库入口与后续内容。

这个顺序的核心原则是：先建立外部读者对问题、立场和价值的理解，再落到系统构成与仓库入口，而不是反过来。

## README 写作约束

为确保后续 README 不偏离本设计，需遵守以下写作约束：

1. 开篇必须把 `AIM` 写成独立产品，而不是模板演化说明。
2. 必须优先面向决策者视角写“为什么需要”，而不是优先给出开发命令。
3. 必须把“人的认知带宽先成为瓶颈”作为问题定义的中心前提之一。
4. 必须明确价值排序以吞吐为先，不得把多个价值点并列为无主次的列表。
5. 必须清晰声明与 `OpenCode` 的 API-only 边界。
6. 必须写清楚 `AIM` 的有观点立场与反泛化取舍。
7. 必须保留 `CZ-Stack` 作为来源说明，但不能让其压过 `AIM` 主叙事。
8. 开发者安装、仓库目录、命令等内容只能放在后部，作为补充入口，不得抢占前半部分主叙事。

## 对后续实现的影响

虽然本次不改 README 正文，但该设计会直接约束后续若干实现动作：

1. 根 README 需要在后续改写中完成主叙事切换。
2. 现有 `CZ-Stack` 模板说明需要让位到单独文档承载。
3. `AIM` 的后续 API、CLI、GUI 与调度能力描述，都需要与本设计中的 MVP 与非目标边界一致。
4. 后续任何面向外部的项目介绍，都应优先复用本设计中的问题定义、定位与价值排序，而不是重新发散产品叙事。

## 风险与边界说明

当前方向已在对话中达成一致，本次无阻塞级待确认项。

需要明确的唯一边界是：本设计用于指导 README 重写与项目目标表达，不自动等价于所有底层实现细节已经定案。只要后续实现仍满足以下约束，就视为符合本设计：

1. `AIM` 被持续表达为独立产品。
2. 系统边界保持 API-only `OpenCode` 集成。
3. 产品主线保持“围绕基线推进的多 Agent 调度与闭环”。
4. 初始范围、路线图与非目标没有被 README 擅自扩张。
