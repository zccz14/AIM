# 在任务看板中补齐前端直连 API 的创建 Task 主路径

## Assumptions

- `modules/web` 当前已经有一个以任务列表为真相源的只读 dashboard，并且同一份任务数据会被概览区、依赖图、表格和详情 drawer 共同消费。
- 当前前端已经具备统一的服务端地址配置和契约 client 包装层，因此新增创建能力应继续直连既有 Task API，而不是引入新的 BFF、表单代理层或本地持久化。
- Task 契约中的创建请求允许 `task_spec` 之外的可选字段，但本次已确认的 GUI 创建范围只需要提交 `task_spec`，不要求用户在首版里填写依赖、session、worktree 或 PR 信息。
- 任务详情已经通过共享 drawer 承载“从看板钻取到单任务”的阅读路径，因此创建成功后的下一步不应发明新的详情承载方式，而应复用现有单任务查看入口。
- 当前 dashboard 中任务的命名来源仍是 `task_spec`，因此后续创建路径也必须保持“标题只从 `task_spec` 派生”这一约束，而不是把标题升级为新的独立字段。

## Goal vs Non-Goal

### Goal

- 在任务看板顶部提供一个明确的主操作入口，让用户一进入 dashboard 就能发现“创建 Task”是当前页面支持的唯一写操作。
- 点击主操作后打开专用创建 drawer，在不离开当前 dashboard 上下文的前提下完成 Task 创建，避免把首版写路径拆成新页面或多步导航。
- 创建表单首版只要求填写 `task_spec`，并把它作为唯一必填事实直接提交到既有 Task API，形成最小、可联调、可继续扩展的 GUI 创建闭环。
- 创建成功后立刻回到 dashboard 的单一事实源：关闭创建 drawer、刷新任务列表查询，并自动打开新建任务的详情 drawer，让用户马上看到新资源已进入当前看板。
- 列表、概览快捷入口、依赖图节点和详情 drawer 对新任务继续沿用同一种标题语义：展示从 `task_spec` 提炼出的首行摘要或截断文本，而不是额外维护独立标题输入。

### Non-Goal

- 不把本次创建入口扩展为完整 Task 编辑器；首版不支持在创建时填写依赖、状态、session、worktree、PR 链接或其他高级字段。
- 不新增独立的创建页面、向导式多步骤流程、内联表格新增行或浮层之外的第二套创建入口。
- 不改变后端 Task 契约、服务端校验语义或 dashboard 之外的任务写接口边界；本次只是让前端开始消费既有创建 API。
- 不在本次顺手补齐任务编辑、删除、批量创建、草稿暂存、自动补全依赖或其他后续写操作能力。
- 不把“列表标题显示什么”从 `task_spec` 派生逻辑改造成新的显式字段，也不要求后端返回额外 title 字段来服务本次创建流程。
- 不为了创建成功后的定位体验而引入新的全局状态体系、路由跳转体系或第二套详情面板。

## Core Path

当前 dashboard 的核心价值是把“任务集合”作为单一事实源来承载概览、关系和单任务钻取，但它仍停留在只读观察面，用户若要从 GUI 发起新的 Task，还缺少一个与现有阅读路径对齐的写入口。本次增量的目标不是重做看板，而是在这个已有事实源之上补出最短创建路径，让 dashboard 从“只能看”变成“可以在原地创建并立刻继续看”。

主路径应保持为一个单上下文闭环。用户先在 dashboard 顶部看到唯一且明确的主按钮，再通过该按钮打开专用创建 drawer。之所以选择 drawer，而不是新页面、内联编辑或 modal 套 modal，是因为 dashboard 本身已经围绕右侧详情 drawer 建立了“留在当前页面内完成任务钻取”的交互预期；创建也应延续这种局部进入、局部退出的节奏，避免用户在创建时丢失当前列表和关系视角。

在创建 drawer 内，首版只暴露 `task_spec` 一个必填输入。这里的设计重点不是把契约的全部可选字段都暴露给用户，而是明确首版的 GUI 创建语义等于“提交一个新的任务说明”。契约中已有的 dependencies、session、worktree、pull request URL、status 等字段继续存在于系统模型中，但在本次主路径里都属于可延后的复杂度，不应抢占入口心智，也不应把创建路径膨胀成高级表单。

提交成功后，系统必须立即回到 dashboard 的统一事实源，而不是停留在一个“提交成功”的孤立提示状态。正确的后续顺序是：关闭创建 drawer，刷新 dashboard 的任务查询，让新任务进入列表与依赖关系的共享数据集，然后自动打开该新任务的详情 drawer。这样用户无需手动再搜索一次，也无需在多个视图之间确认“任务到底有没有创建出来”，而是直接进入“刚创建的任务现在是什么状态”的下一步观察动作。

标题语义必须在创建前后保持一致。Task 在 dashboard 中被识别的名称仍然应是从 `task_spec` 派生出来的首行摘要或截断文本，而不是创建时额外输入一个 title 再在展示层分叉维护。这样可以保证列表、概览入口、依赖图节点和详情 drawer 对同一任务保持同一命名来源，避免出现“创建表单里的标题”和“任务实际说明”长期漂移的问题。

备选路径有几条，但都不应采用。第一条备选路径是做独立创建页面或路由跳转，这会让首版创建脱离 dashboard 的单上下文体验，并引入本次不需要的导航复杂度。第二条备选路径是把所有可选字段一次性铺进创建表单，看似更完整，实则会把首版最小 GUI 创建目标升级成通用任务编辑器。第三条备选路径是创建成功后只弹出提示、不自动打开详情，这会把“我刚创建了什么”重新变成用户的手动检索成本。相比之下，顶部主按钮 + 专用创建 drawer + 成功后刷新并自动钻取详情，能以最小交互面完成最完整的前端创建闭环。

## Value Alignment

- 当“尽快提供一个稳定可用的 GUI 创建闭环”和“首版就暴露更多高级字段”冲突时，优先前者；因此表单只要求 `task_spec`，其余契约字段暂不进入创建入口。
- 当“保持 dashboard 仍是单一工作上下文”与“为创建单独开页面或新导航层级”冲突时，优先前者；因此创建动作应以内聚的 drawer 形式挂在现有看板之内。
- 当“创建后立即确认新任务已经进入当前事实源”与“只给出抽象成功提示”冲突时，优先前者；因此成功后应刷新列表并自动打开新任务详情。
- 当“维持任务命名语义单一稳定”与“顺手新增一个独立 title 字段让表单更像传统表单”冲突时，优先前者；因此继续以 `task_spec` 的首行摘要或截断文本作为标题来源。
- 当“让首版 scope 保持为前端直连既有 API 的最小增量”与“顺手改后端契约、状态模型或更多写操作能力”冲突时，优先前者；因此本次只补前端创建路径，不扩展到更大的 Task 编辑面。
