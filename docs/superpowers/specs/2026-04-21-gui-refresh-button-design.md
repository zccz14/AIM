# Dashboard 页头 Refresh 按钮设计说明

## 假设

- 当前 dashboard 页面已经通过 React Query 统一读取任务看板数据，并且页头存在 `Create Task` 主操作，页面错误态已包含可点击的 `Retry` 入口。
- 当前页面的筛选输入、抽屉开关和选中任务等交互状态由本地 UI 状态驱动，而不是依赖整页浏览器刷新才能恢复。
- 错误态 `Retry` 与正常态页头按钮本质上都在表达“重新拉取当前 dashboard 数据”，因此可以收敛为同一刷新动作，而不需要维护两套刷新语义。
- 现有 React Query 已能暴露进行中刷新状态，足以驱动按钮 loading / disabled 表现；本次不需要为刷新单独增加新的全局状态层。
- 若刷新后当前选中的任务仍存在，页面依旧可以基于更新后的共享数据继续展示该任务；若任务不存在，当前页面已经存在既定降级行为，本次不改写该分支。

## Goal vs Non-Goal

### Goal

- 在 dashboard 页头中、紧邻 `Create Task` 的位置新增一个 `Refresh` 按钮，让用户在正常浏览状态下也能显式触发数据刷新。
- Refresh 只重新请求 dashboard 数据，不触发浏览器整页刷新，从而保留当前筛选条件、抽屉开合和其他本地 UI 状态。
- 在 `dashboard-page.tsx` 中收敛一个统一的 `handleRefresh`，同时供页头 `Refresh` 和错误态 `Retry` 调用，保持刷新入口行为一致。
- 复用现有 React Query 刷新状态驱动 `Refresh` 按钮的 loading 与 disabled，避免用户在刷新进行中重复触发请求。
- 为该交互补充一个 Playwright 验证，确认点击页头 `Refresh` 会触发新的数据请求，并保留用户已输入的筛选内容。

### Non-Goal

- 不引入浏览器级 `window.location.reload()`、路由重载或其他整页重置手段。
- 不重构 React Query 查询层、不改造 query key 体系，也不把本次需求扩展成通用数据同步框架。
- 不新增 toast、全局提示或新的错误呈现方式；刷新失败时继续沿用当前页面已有错误 UI。
- 不改变“选中任务在刷新后已不存在”时的既有回退逻辑，只要求保持现状不退化。
- 不在本次顺手扩展额外刷新入口、自动轮询或跨页面同步能力。

## Core Path

当前 dashboard 已经提供任务查看、筛选和详情抽屉，但在正常成功态中缺少一个显式的“重新拉取最新数据”入口。用户若想确认列表、概览和详情是否已更新，不能依赖浏览器整页刷新，因为整页刷新会把当前筛选输入、抽屉开合与上下文一起重置，破坏 dashboard 作为单页工作面的连续性。本次设计的目标就是补上这个最小刷新入口，同时保持页面的局部状态不被打断。

主路径应当非常直接：用户在页头 `Create Task` 旁边看到 `Refresh` 按钮，点击后仅触发 dashboard 数据重新获取。刷新期间，按钮基于既有 React Query 状态进入 loading 和 disabled，明确提示“请求正在进行”，同时阻止重复点击制造并发请求。刷新完成后，页面上的摘要、表格、图和详情都从同一份更新后的 dashboard 数据重新渲染，而不是通过浏览器 reload 重建整个页面。

为了避免正常态刷新与错误态重试语义分叉，页面需要在 `dashboard-page.tsx` 中维护单一的 `handleRefresh`。页头 `Refresh` 按钮与错误态 `Retry` 都只调用这一个动作。这样后续无论是正常浏览时主动拉取，还是错误后再次尝试，底层都走同一条“重新获取 dashboard 数据”的路径，避免一个入口走 query refetch、另一个入口走独立逻辑而逐渐漂移。

本次对局部状态的要求是“刷新数据，不重置界面”。因此筛选输入框中的已输入文本必须在刷新后保留，已打开的抽屉也不应因刷新动作被主动关闭。如果当前选中的任务在新数据中依然存在，详情区域应自然显示该任务的最新数据；如果该任务已不存在，则继续沿用当前页面已存在的 fallback 行为，不额外发明新提示或新状态机。

备选实现中最容易越界的是把刷新做成查询层重构，或者顺手新增 toast 提示与更多同步机制。这些都不是当前需求的核心。相较之下，在页头增加一个显式按钮、让它与错误态 `Retry` 共用 `handleRefresh`、并完全复用现有 React Query 状态，是满足需求且风险最小的方案。

## Verification Scope

- 增加一个 Playwright 用例，覆盖用户在 dashboard 页输入筛选内容后点击页头 `Refresh` 的路径。
- 用例需要确认点击 `Refresh` 后产生新的 dashboard 请求，而不是仅复用已有渲染结果或触发浏览器整页刷新。
- 用例需要确认刷新完成后，筛选输入中的既有文本仍然保留，证明本地 UI 状态未被重置。

## Value Alignment

- 当“用户需要一个可靠的手动刷新入口”和“尽量不改动现有页面结构”冲突时，优先前者，但实现仅限在页头补一个最小按钮。
- 当“刷新后拿到最新 dashboard 数据”和“保留当前筛选与抽屉上下文”冲突时，优先两者同时满足；因此只能做数据级 refetch，不能做整页 reload。
- 当“统一维护刷新语义”与“为 Retry 和 Refresh 分别写一套逻辑”冲突时，优先前者；因此收敛单一 `handleRefresh`。
- 当“快速落地按钮能力”与“顺手重构 query 层”冲突时，优先前者；因此明确禁止查询层重构。
- 当“刷新失败反馈保持一致”与“再新增 toast 或额外错误表现”冲突时，优先前者；因此继续沿用现有页面错误 UI。
