# Server 全局开放 CORS 设计说明

## 背景 / 当前状态

当前 `modules/api/src/app.ts` 只负责创建顶层 `Hono` 应用、注册路由并暴露 `/openapi.json`，尚未注册显式 CORS 中间件。这意味着现有跨域行为依赖运行环境默认值，仓库内没有一个统一、可审计的“允许任意来源访问全部端点”的声明式配置。

本次用户要求是：Server 在所有环境下都不控制 CORS，生产期也保持全部端点允许任意跨域访问，不引入 origin 白名单。

## 目标

1. 在 API 顶层统一注册一次 CORS 中间件，使当前与未来端点默认继承同一跨域策略。
2. 在所有环境中都允许任意来源访问，不区分开发、测试、生产。
3. 保持方案简单直接，不为本次需求增加环境分支、白名单配置或按路由单独配置。

## 选定方案

在 `modules/api/src/app.ts` 的顶层 `Hono` 应用上注册一次 Hono 官方 CORS 中间件，并将 `origin` 配置为通配语义，使其对整棵路由生效。

选择该方案的原因：

1. 改动点最小，直接贴合“全端点任意 CORS”的需求。
2. 配置集中在应用入口，当前路由与后续新增路由都会自动继承。
3. 不需要为不同环境或不同端点维护额外策略，避免后续出现配置漂移。

## 预期行为

1. 所有 API 端点（包括当前已注册路由与后续新增路由）都返回允许任意来源访问的 CORS 响应头。
2. 浏览器发起的跨域预检请求能够按同一全局策略通过。
3. 开发、测试、生产环境保持同一行为，不存在仅开发期开放、生产期收紧的分支逻辑。
4. 不引入任何 origin 白名单、环境变量开关或按路由例外。

## 非目标

1. 不调整鉴权、会话、Cookie 或其他安全策略。
2. 不为特定端点设置更细粒度的 CORS 差异化规则。
3. 不新增配置文件、运行时开关或管理界面来控制跨域策略。
4. 不改动除顶层 CORS 注册之外的路由组织方式。

## 风险与注意事项

1. 全局开放 CORS 会明确允许任意第三方前端直接从浏览器调用这些端点；这是本次用户明确接受的行为，而不是附带副作用。
2. 若未来引入依赖浏览器凭据语义的接口，需要重新审视“任意来源”与凭据策略的兼容性；本次不预留兼容层。
3. 该策略会同时作用于当前与未来端点，因此后续若出现需要受限访问的接口，必须显式调整本设计，而不是假设默认行为会更严格。

## 最小验证

1. 检查 `modules/api/src/app.ts`，确认 CORS 中间件在顶层应用注册一次，且作用范围覆盖全部路由。
2. 确认配置表达的是任意来源访问，而非环境相关分支或白名单。
3. 以最小必要方式验证普通跨域请求与预检请求都能得到预期 CORS 响应头。
